## 60-90 秒版话术

> 这个项目是我自己做的一个企业智能代理系统，背景来自我实习期间真实遇到的排障痛点。传统排障流程里，收到告警之后，通常要先去看 Prometheus 指标，再去日志平台查日志，还要翻内部文档找处理方案，整个过程比较碎，而且信息收集成本很高。
>
> 所以我做了一个原型系统，把知识问答和运维排障两个场景整合在一起。对话场景下，我基于 Spring AI Alibaba 的 ReactAgent 实现了 ReAct 模式，让系统可以自动决定什么时候调用内部文档检索、告警查询、日志查询这些工具，并支持多轮会话和 SSE 流式输出。最近我又把 RAG 检索链路升级成了两阶段方案，先用 Milvus 做向量粗排，再用 rerank 模型做精排，同时通过单文档 chunk 限流减少重复召回。运维场景下，我设计了 Supervisor、Planner、Executor 的多 Agent 工作流，让系统能够自动完成任务规划、工具执行、结果汇总，并输出结构化告警分析报告。
>
> 整个系统后端用 Spring Boot + Java 17，前端是 React + TypeScript，RAG 链路里用了 DashScope 向量化和 Milvus 检索。这个项目对我来说最有价值的地方，是我把 Agent、RAG、工具调用和前端交互完整串起来了，而不只是做了一个简单的聊天 Demo。

## 3 分钟版话术

> 这个项目的出发点，是我在实习期间经常遇到的一个问题：收到线上告警之后，通常要跨多个系统来回切换，先查监控，再查日志，再翻内部文档，最后再自己整理结论，这个过程比较耗时，而且很多步骤本质上是在重复做信息收集。
>
> 所以我想做一个企业智能代理原型系统，把知识问答和运维排障两个场景整合起来。一方面，它可以基于内部文档做 RAG 问答；另一方面，它可以围绕告警场景自动执行分析流程，最后输出一份结构化的排障报告。
>
> 技术实现上，我把这个项目拆成了四块。第一块是对话 Agent，我基于 Spring AI Alibaba 的 ReactAgent 做了 ReAct 模式，让模型可以根据问题自主决定是否调用内部文档检索、Prometheus 告警查询、日志查询这些工具。第二块是 RAG 链路，我实现了文档上传、分片、向量化、Milvus 存储和语义检索的完整流程，最近又把它从单阶段向量 topK 升级成了 Milvus 粗排加 rerank 精排的两阶段检索。第三块是 AIOps 多 Agent 编排，我把角色拆成了 Supervisor、Planner、Executor，分别负责调度、规划和执行，避免一个 Agent 同时承担所有职责。第四块是前端交互，我用 React + TypeScript 做了流式对话、会话管理、文件上传和运维分析界面，把后端能力完整地承接出来。
>
> 这个项目里我最想强调的不是“我会调模型接口”，而是我把一个相对真实的业务问题拆解成了可以落地的技术方案，并完成了从后端 Agent 编排到前端交互的完整闭环。它目前是一个个人原型项目，但已经能比较完整地展示我在后端架构、AI 应用落地和工程整合上的能力。

3. `多 Agent 编排`
不是单一 Agent 生硬堆 prompt，而是拆成规划、执行、调度几个角色。

4. `前后端完整闭环`
不是只有后端 demo，还配了流式交互页面、会话切换、文件上传和运维分析弹窗，能把整个系统完整跑起来。

## 最近新增能力怎么讲

如果面试官问“你最近还做了哪些优化”，优先讲这 3 个：

1. `两阶段检索`
我把原来的一阶段向量检索升级成了两阶段方案，先由 Milvus 做高召回粗排，再由 rerank 模型做精排，提升最终送给大模型的上下文质量。

2. `文档级限流`
我发现纯向量检索容易把同一篇文档里相邻的多个 chunk 一起召回，所以又加了单文档 chunk 上限，避免最终上下文被单篇文档占满。

3. `检索链路可观测性`
我补了粗排候选、rerank 分数和最终命中的日志，方便分析是粗排没召回到，还是精排排序不理想，这样后面调 chunk、topK、nprobe 才有依据。

## RAG 实验这块怎么讲

这部分不要只讲一句“我测过 85%”，那样很容易被追问穿。

你应该把它讲成一个完整的小型离线实验：

1. 我先把语料分成三层
   - 核心文档：主 runbook，最完整、最标准的处理文档
   - 同主题变体文档：FAQ、SOP、事故复盘、附录
   - 相似干扰文档：主题接近但回答角度更细或不同

2. 我再设计 query 集
   - keyword
   - paraphrase
   - step
   - detail

3. 然后对比两组方案
   - 纯向量粗排：`Milvus top3`
   - 两阶段检索：`Milvus top12 -> rerank -> top3`

4. 指标主要看
   - `Hit@1`
   - `Hit@3`
   - `MRR`
   - `Avg Latency`

5. 评测时我用了两种口径
   - 严格口径：只把强相关文档算命中
   - 宽松口径：只要能支持回答的问题相关文档就算命中

这段的核心价值是：

> 我不是随便报一个数字，而是把语料组织、query 设计、评分口径和对比组都做清楚了。

### 核心文档和非核心文档怎么解释

推荐说法：

> 核心文档可以理解成主 runbook，是这个主题下最完整、最标准的处理指南。非核心文档不等于错误文档，它包括 FAQ、SOP、事故复盘和某些更细分的专题文档。比如用户问的是步骤型问题，SOP 可能比主 runbook 更适合；如果问的是 Redis BigKey 超时，那一个专题文档可能比慢响应总文档更适合。所以我后面做评测时，不能只看有没有回到核心文档，还要看有没有找到能真正支持回答的相关文档。

### 实验里最能体现你做过的一个点

推荐主动讲这个：

> 我一开始其实踩过一个坑。最早那版评测脚本只按单个 target_doc 计分，没有把 FAQ、SOP 和高相关专题文档纳入评分，导致我一度得出了“rerank 反而变差”的错误结论。后来我复盘数据，发现问题出在评分口径，而不是检索本身，所以我把评测改成了 label-aware 的方式，按 relevance level 重新计算。这个过程让我意识到，RAG 实验里评测口径本身和检索模型同样重要。

这段很值钱，因为它体现的是：

- 你真的做了实验
- 你不是机械跑脚本
- 你能发现实验设计本身的问题

## 高频追问与回答模板

### Q1：这个项目为什么要做？

推荐回答：

回答重点：

- 背景来自真实场景
- 不是纯技术炫技
- 目标是自动化信息收集和初步分析

---

### Q2：为什么不是普通聊天机器人，而要做 ReAct？

推荐回答：

> 因为这个场景不是简单地“生成一句答案”，而是经常需要先判断要不要查文档、查指标、查日志。普通 prompt 模式更适合直接回答问题，但不适合多工具组合场景。ReAct 更适合“思考-行动-观察”的闭环，所以我在对话场景里用了 ReactAgent。

回答重点：

- 普通聊天和任务型问答的区别
- ReAct 的核心价值是工具调用决策
- 你不是为了用新词，而是因为场景需要

---

### Q3：你的 RAG 链路完整流程是什么？

推荐回答：

> 文档上传之后，我先做分片，再调用 embedding 模型生成向量，存到 Milvus。用户提问时，会先做向量检索拿到一批候选，再通过 rerank 模型做二次精排，最后只把少量高质量 chunk 和用户问题一起交给大模型生成答案。为了减少重复信息，我还加了单文档 chunk 上限，避免同一篇文档的相邻分片占满上下文。也就是说，RAG 这部分我做的是从入库到召回再到精排的完整链路，而不是只做了一个检索接口。

回答重点：

- 上传
- 分片
- 向量化
- 向量库存储
- 粗排检索
- rerank 精排
- 文档级限流
- 生成

---

### Q4：为什么要做多 Agent，而不是一个 Agent 搞定？

推荐回答：

> 因为 AIOps 排障是一个多步骤任务。它不是单轮问答，而是先规划，再执行，再根据执行结果调整下一步。如果一个 Agent 同时负责规划和执行，职责会比较混乱，也不容易控制流程。所以我把角色拆成了 Supervisor、Planner、Executor，分别负责调度、规划和执行，这样职责更清晰，也更适合后续扩展。

回答重点：

- 单 Agent 适合简单任务
- 多步骤排障更适合角色拆分
- 多 Agent 的核心不是“更高级”，而是“职责更清晰”

---

### Q5：你封装了哪些工具？模型怎么知道什么时候调用？

推荐回答：

> 我封装的核心工具主要有三类：内部文档检索、Prometheus 告警查询、日志查询。另外还有时间工具这种辅助能力。模型什么时候调用工具，主要依赖两部分：一部分是系统提示词里对工具适用场景的约束，另一部分是工具本身的描述信息。模型会根据用户问题判断是否需要查资料或者查监控，而不是每次都盲目调用。

回答重点：

- 工具要分能力边界
- 调用不是写死 if-else
- 但也不是完全放飞，还是通过 prompt 和描述来约束

### Q6：你说 `TopK 命中率 85%`，这个怎么来的？

推荐回答：

> 这个数据不是我拍脑袋说的，而是我基于现有知识库做的一版小规模离线检索实验。实验里我没有只放核心 runbook，还补了 FAQ、SOP、事故复盘和一些相似干扰文档，让它更接近真实知识库。query 也不是一种类型，我分成了关键词直问、口语化改写、步骤型问题和细节型问题。
>
> 评测时我主要看 `Hit@1`、`Hit@3` 和 `MRR`。其中 `Hit@3` 是最核心的，因为它能反映 Top3 里是否至少出现了一个相关文档。后来我又意识到，一个 query 并不一定只有一个“唯一正确文档”，有些问题其实更适合命中 FAQ 或 SOP，所以我把评测改成了 label-aware 的方式，给每个 query 标注强相关和弱相关文档，再分严格口径和宽松口径去算。  
>
> 所以我现在更稳的说法不是“我有一个绝对准确的 85%”，而是：我做过离线评测，并且把语料组织、query 设计、评分口径和对比组都做了拆分，能够比较清楚地判断检索优化到底有没有带来提升。

更安全的短版：

> 这是我基于现有知识库做的离线小样本评测结果，不是正式线上指标，但评测方法是成体系的。

回答重点：

- 一定强调 `离线实验`、`对比组`、`评分口径`
- 不要只报数字，要讲“怎么测的”
- 你要表现得像自己真的做过评测设计和复盘

### Q7：你为什么要区分核心文档和非核心文档？

推荐回答：

> 因为 RAG 检索不一定总是“回到主文档”才算好结果。核心文档是主 runbook，适合回答比较完整、比较标准的问题；但如果用户问的是步骤型问题、FAQ 型问题，或者某个更细分的专题问题，比如 Redis 超时、连接池耗尽，那 FAQ、SOP 或专题文档反而更适合。所以我评测时要区分两件事：第一，系统有没有回到核心文档；第二，它有没有找到能真正支撑回答的相关文档。这两个目标不能混在一起看。

回答重点：

- 核心文档是主 runbook
- 非核心文档不等于错误文档
- 你在区分“主文档路由”和“回答支持型检索”

---

### Q8：你为什么把 RAG 从一阶段检索升级成两阶段检索？

推荐回答：

> 因为我在实际调试时发现，纯向量 topK 更像是粗召回，能找到“可能相关”的内容，但不一定能把最适合回答问题的 chunk 排在最前面。尤其是在运维文档场景里，同一篇文档里相邻 chunk 很容易一起被召回，导致上下文里重复信息偏多。所以我把检索拆成了两阶段，先用 Milvus 做高召回粗排，再用 rerank 模型做精排，最后再通过单文档上限控制冗余。

回答重点：

- 一阶段向量检索不是错，但排序不够细
- 两阶段的目标是“排准”，不是只为了堆复杂度
- 你是根据实际问题迭代出来的，不是照着论文堆架构

---

### Q9：你做实验时有没有踩过坑？后来怎么修正的？

推荐回答：

> 有，而且这个坑对我帮助很大。最开始我用的是比较简单的评测方式，只按 query 对应的单个 target_doc 来计分。后来我复盘结果时发现，很多明明很合理的命中文档，比如 FAQ、SOP，都会被算成错。这个时候我才意识到问题不在 rerank 本身，而在评分口径。所以我后面补了 labels，把强相关和弱相关文档都标出来，再按严格口径和宽松口径分别算指标。修正以后，两阶段检索的 Hit@3 和 MRR 是明显优于纯粗排的，但延迟也确实变高了。

回答重点：

- 主动承认实验设计踩坑
- 说明你能复盘并修正口径
- 最后落到“效果提升 + 延迟代价”这个更真实的结论

---

### Q10：为什么还要加 `per-doc-cap` 这种单文档上限？

推荐回答：

> 因为 chunk 做了分片之后，同一篇文档的相邻内容语义往往很接近，向量检索和 rerank 都可能把它们排得很靠前。如果不限制，最终上下文可能被单篇文档的多个相邻 chunk 占满，信息覆盖面反而下降。所以我加了一个单文档上限，让最终上下文既相关又有一定多样性。

回答重点：

- 它解决的是“重复召回”，不是普通去重
- 目标是提升上下文覆盖面
- 这是一个很工程化的小优化点

---

### Q11：你说“排障时间从小时级降低到分钟级”，这个怎么证明？

推荐回答：

> 这个表述我会更谨慎一些。它不是严格线上 A/B 数据，而是基于我实习时人工排障流程的经验对比。传统流程里，SRE 要自己去多个系统收集信息，通常要 30 分钟甚至更久；而这个原型系统可以在几分钟内把监控、日志、文档相关的信息自动整理出来，并生成一份结构化分析报告。所以更准确的说法是，它显著缩短了排障前置信息收集和初步分析的时间。

最稳的一句收口：

> 这是原型收益判断，不是正式线上量化结论。

回答重点：

- 先主动降预期
- 再强调它节省的是“前置信息收集成本”
- 不要硬扛成“我证明了效率提升多少倍”

### Q12：这个项目最大的难点是什么？

推荐回答：

> 我觉得最大的难点有两个。第一个是工具调用可靠性，因为模型不一定每次都能正确判断该查什么、传什么参数；第二个是多 Agent 之间的职责边界和上下文传递，因为 Planner 和 Executor 如果接口不清晰，后面的再规划就会失效。也就是说，这个项目真正难的地方不是把模型接进来，而是怎么让它围绕真实任务稳定工作。

回答重点：

- 不要回答成“技术栈好多”
- 要回答成“任务稳定性和流程设计”

### Q13：如果继续做下去，你下一步会优化什么？

推荐回答：

> 如果继续迭代，我会优先补三块。第一是分片策略升级，比如从单一 Markdown 分片扩展到对 PDF 的版面清洗和结构化分片；第二是把评测体系继续细化，特别是把“回答支持型检索”和“核心 runbook 路由”拆成两套指标，不再混在一起；第三是继续做性能优化和降级策略，比如降低 rerank 延迟，或者在粗排置信度足够高时跳过 rerank。因为当前版本更偏原型验证，下一步应该往工程化方向补齐。

回答重点：

- 评测体系
- 降级兜底
- 工程化能力

### Q14：前端是你自己写的吗？

推荐回答：

回答重点：

- 强调“够用且完整”
- 不要把重点放在 UI 组件堆砌
