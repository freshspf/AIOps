## 一句话定位

SuperBizAgent 是我围绕真实排障痛点做的一个企业智能代理原型系统，核心是把 `RAG 检索`、`工具调用`、`多 Agent 编排` 和 `流式交互` 串成完整闭环，用来解决内部知识问答和告警排障两个典型场景。最近一轮迭代里，我把 RAG 从基础向量召回升级成了 `Milvus 粗排 + rerank 精排` 的两阶段检索方案。

## 起手介绍项目

**这个项目本质上是一个企业内部的 AI 助手系统，主要想解决两个问题。**

第一个是让业务或者研发能基于内部文档做智能问答，减少靠人查资料的成本；

第二个是针对线上告警和故障排查，做一个 AI 辅助分析流程，提升问题定位效率。

这个项目的背景是我们当时遇到过一次线上事故，排查花了很久，所以我就想结合 RAG 和 Agent 的方式，做一个偏 AIOps 场景的智能助手。



**整体上我把它分成两条链路。**

第一条是 RAG 问答链路，先对内部 markdown 文档做切分、向量化和入库，用户提问后做向量召回，再经过 rerank 精排，把最相关的上下文交给大模型生成答案。

第二条是 AIOps 链路，因为故障分析不是单步问答，而是多步骤任务，所以我没有继续用单 Agent，而是拆成了 Planner、Executor 和 Replanner 这样的多 Agent 协作流程，让它能先规划、再执行、再根据结果调整计划。



**我主要做了几块设计。**

第一块是 RAG 检索链路，包括文档切分、chunk 参数、向量检索加 rerank 的两阶段召回。

第二块是多轮对话上下文管理，我用 Redis 持久化历史消息，并维护最近几轮对话加摘要，避免上下文无限增长。

第三块是 AIOps 的多 Agent 编排，把日志查询、文档检索这些能力封装成工具，让 Agent 能按步骤调用。

整体上这个项目更偏一个可运行的 MVP，但核心链路是完整的，重点是把问答和故障分析这两类场景拆开设计，而不是用一套方案硬套所有问题。

## 项目介绍话术

推荐直接这样讲：

> 这个项目是我自己做的一个企业智能代理原型系统，核心想解决的是企业内部知识问答和运维排障两个场景。背景来自我实习时观察到的真实问题，就是收到告警以后，通常要在监控、日志和内部文档之间来回切换，信息收集成本很高。  
>
> 所以我做了一个统一系统，把知识库问答、工具调用、AIOps 多 Agent 编排和前端交互串起来。对话场景下，系统能基于 ReAct 模式自主决定什么时候查文档、查指标、查日志；运维场景下，我把流程拆成 Supervisor、Planner、Executor 三个角色，完成告警读取、任务规划、工具执行和报告生成。  
>
> 最近我还把 RAG 从基础向量召回升级成了两阶段检索，也就是 Milvus 粗排加 rerank 精排，并补了一套离线检索评测，验证检索优化是否真的有效。这个项目对我来说最重要的，不是接了模型接口，而是把一个真实业务问题拆成了可落地的工程方案。

## 业务流程话术

面试官如果问“这个系统实际是怎么跑起来的”，可以这样讲：

> 整个业务流程可以分成两条主线。第一条是知识问答：用户提问后，Agent 先判断需不需要调用工具，如果需要，会走内部文档检索；检索链路里先做向量粗排，再做 rerank 精排，最后把高质量上下文交给大模型生成答案，并通过 SSE 流式返回前端。  
>
> 第二条是运维排障：系统先读取告警，再由 Supervisor 调度 Planner 和 Executor。Planner 负责拆任务，比如先查 Prometheus 告警、再查日志、再结合文档分析；Executor 负责执行当前第一步并回传结构化反馈；Supervisor 再决定是继续规划、继续执行，还是结束并输出报告。  
>
> 这两条线共用一套工具体系和会话体系，所以项目看起来是知识库、对话、运维三个能力，但底层其实是一套统一的 Agent + RAG + 工具调用框架。

## 简历要点逐条话术

下面按 [简历写法.md](/Users/joer/Gitroom/SuperBizAgent/简历以及面试话术/简历写法.md) 里的要点准备。

### 要点1：基于 Spring AI Alibaba 设计并实现企业内部知识问答 Agent 与 智能运维 Agent

这句话实际在说什么：

你不是只做了一个聊天入口，而是做了两类 Agent：一个偏知识问答，一个偏任务执行。

难点：

- 两类场景任务形态不同
- 问答 Agent 更偏 retrieval + generation
- 运维 Agent 更偏 planning + execution

为什么这么设计：

因为知识问答和故障排查不是一类问题。如果全塞进一个万能 Agent，prompt 会很重，边界也不清晰。

面试官可能会问什么：

- 为什么不做成一个万能 Agent？
- Spring AI Alibaba 你主要用了什么能力？
- 这两个 Agent 的差别是什么？

推荐回答：

> 我没有把所有能力堆到一个 Agent 里，因为知识问答和运维排障的任务形态差异很大。问答 Agent 主要解决“查资料并回答”，运维 Agent 主要解决“拆任务、调工具、回收结果”。我用 Spring AI Alibaba 的 Agent 能力做了这两个方向的封装，本质上是在同一个框架里做不同职责的 Agent，而不是做一个提示词特别重的万能机器人。

### 要点2：基于 ReAct 模式实现对话 Agent，支持工具自动调用、多轮会话记忆与 SSE 流式输出

这句话实际在说什么：

你做的是一个任务型问答 Agent，不是普通聊天框。

难点：

- 工具什么时候调用
- 多轮上下文怎么恢复
- SSE 流式和会话持久化怎么配合

为什么这么设计：

因为很多问题不能靠模型直接编，需要先查文档、查日志、查指标。ReAct 更适合这种“思考-行动-观察”的闭环。

面试官可能会问什么：

- 为什么是 ReAct，不是普通 prompt？
- 会话记忆现在怎么做？
- 为什么选 SSE？

推荐回答：

> 我这里选 ReAct 主要是因为场景不是单纯生成一句回答，而是经常需要先判断要不要调用工具。多轮记忆这块我现在做的是 Redis 持久化加窗口恢复，长历史做摘要压缩；SSE 则是为了降低交互等待感，让前端能流式显示中间结果和最终回答。

### 要点3：设计并落地 RAG 知识库链路，实现分片、向量化、Milvus 存储与语义检索；经过测试验证，TopK命中率达到85%

这句话实际在说什么：

你做的是完整 RAG，不只是“向量库查一下”。

难点：

- chunk 怎么切
- 检索怎么调
- 评测怎么做才不虚

为什么这么设计：

因为企业问答依赖内部知识，模型参数本身不够，必须引入检索；而且后面已经升级成两阶段检索，不再是单纯 topK。

面试官可能会问什么：

- 85% 怎么来的？
- 为什么要上 rerank？
- 什么是核心文档和非核心文档？

推荐回答：

> 我最开始做的是基础向量检索，后来升级成了 Milvus 粗排加 rerank 精排。为了验证优化是否真的有效，我做了一版离线评测。语料我分成核心 runbook、同主题变体文档和相似干扰文档三层，query 也分成关键词、口语化、步骤型和细节型。后面我还专门修过评测口径，因为一开始只按单个 target_doc 打分，会把 FAQ 和 SOP 这种合理命中误判成错。修正成 label-aware 之后，两阶段检索在 Hit@3 和 MRR 上是优于纯粗排的。

### 要点4：基于 Supervisor + Planner + Executor 实现 AIOps 工作流，打通告警读取 -> 任务规划 -> 工具查询 -> 结果汇总 -> 报告生成

这句话实际在说什么：

这是你项目里最有区分度的一点，说明你不只是做问答，而是做“任务闭环”。

难点：

- 多 Agent 的职责边界
- Planner 和 Executor 之间状态传递
- 失败时怎么终止，而不是无限循环

为什么这么设计：

因为运维排障天然是多步骤任务，单 Agent 容易既负责规划又负责执行，最后 prompt 非常重，还不好控制失败边界。

面试官可能会问什么：

- 多 Agent 到底怎么编排？
- 为什么不是单 Agent？
- 你这里的 Supervisor 是什么角色？

推荐回答：

> 我这套多 Agent 不是自由协商式的 swarm，而是 Supervisor 驱动的状态机式编排。Planner 负责规划和再规划，Executor 只执行当前第一步，Supervisor 根据共享状态决定下一步是继续规划、继续执行，还是直接结束。这样角色边界比较清晰，也更适合加终止条件和失败控制。

这一条有两句建议不要按原简历硬讲：

- `结合历史工单生成运维建议方案`
- `将运维响应时间从小时级降低到分钟级`

更安全的说法：

> 系统会结合内部文档、日志和监控结果生成处理建议。  
> 从原型验证角度看，它能明显缩短排障前置信息收集和初步分析时间。

### 要点5：封装内部文档检索、Prometheus 告警查询、日志查询等工具能力，并结合 MCP Client 对接外部日志服务

这句话实际在说什么：

你在做的是 Agent 的工具层，不是纯模型层。

难点：

- 工具能力边界怎么定义
- 参数怎么约束
- 外部系统接入怎么保持可替换

为什么这么设计：

因为真实企业场景里，问题不只是“生成答案”，而是要查外部系统的数据。没有工具层，Agent 只能空想。

面试官可能会问什么：

- 工具怎么被 Agent 调起来的？
- MCP 在这里的价值是什么？
- 如果工具失败怎么办？

推荐回答：

> 我把文档检索、Prometheus 告警查询、日志查询这些能力封装成了独立工具，让 Agent 根据问题自主决定是否调用。MCP 的价值主要是把外部系统能力抽象成统一可调用接口，这样日志服务不一定绑死在某一家实现上。后面如果工具失败，我会把错误信息回传给上层流程，而不是让模型继续编结果。

## 项目亮点怎么讲

面试里优先突出这 4 个点：

1. `ReAct + 工具调用`
不是纯聊天，而是能根据问题自主决定要不要查文档、查指标、查日志。

2. `RAG 检索链路完整且做了迭代升级`
从文档上传、分片、向量化、入库，到查询时检索再生成，链路是闭环的；最近又升级成了 `Milvus 粗排 + rerank 精排 + 单文档限流` 的两阶段检索。

3. `多 Agent 编排`
不是单一 Agent 生硬堆 prompt，而是拆成规划、执行、调度几个角色。

4. `前后端完整闭环`
不是只有后端 demo，还配了流式交互页面、会话切换、文件上传和运维分析弹窗，能把整个系统完整跑起来。

## 最近新增能力怎么讲

如果面试官问“你最近还做了哪些优化”，优先讲这 3 个：

1. `两阶段检索`
我把原来的一阶段向量检索升级成了两阶段方案，先由 Milvus 做高召回粗排，再由 rerank 模型做精排，提升最终送给大模型的上下文质量。

2. `文档级限流`
我发现纯向量检索容易把同一篇文档里相邻的多个 chunk 一起召回，所以又加了单文档 chunk 上限，避免最终上下文被单篇文档占满。

3. `检索链路可观测性`
我补了粗排候选、rerank 分数和最终命中的日志，方便分析是粗排没召回到，还是精排排序不理想，这样后面调 chunk、topK、nprobe 才有依据。

## RAG 实验这块怎么讲

这部分不要只讲一句“我测过 85%”，那样很容易被追问穿。

你应该把它讲成一个完整的小型离线实验：

1. 我先把语料分成三层
   - 核心文档：主 runbook，最完整、最标准的处理文档
   - 同主题变体文档：FAQ、SOP、事故复盘、附录
   - 相似干扰文档：主题接近但回答角度更细或不同

2. 我再设计 query 集
   - keyword
   - paraphrase
   - step
   - detail

3. 然后对比两组方案
   - 纯向量粗排：`Milvus top3`
   - 两阶段检索：`Milvus top12 -> rerank -> top3`

4. 指标主要看
   - `Hit@1`
   - `Hit@3`
   - `MRR`
   - `Avg Latency`

5. 评测时我用了两种口径
   - 严格口径：只把强相关文档算命中
   - 宽松口径：只要能支持回答的问题相关文档就算命中

这段的核心价值是：

> 我不是随便报一个数字，而是把语料组织、query 设计、评分口径和对比组都做清楚了。

### 核心文档和非核心文档怎么解释

推荐说法：

> 核心文档可以理解成主 runbook，是这个主题下最完整、最标准的处理指南。非核心文档不等于错误文档，它包括 FAQ、SOP、事故复盘和某些更细分的专题文档。比如用户问的是步骤型问题，SOP 可能比主 runbook 更适合；如果问的是 Redis BigKey 超时，那一个专题文档可能比慢响应总文档更适合。所以我后面做评测时，不能只看有没有回到核心文档，还要看有没有找到能真正支持回答的相关文档。

### 实验里最能体现你做过的一个点

推荐主动讲这个：

> 我一开始其实踩过一个坑。最早那版评测脚本只按单个 target_doc 计分，没有把 FAQ、SOP 和高相关专题文档纳入评分，导致我一度得出了“rerank 反而变差”的错误结论。后来我复盘数据，发现问题出在评分口径，而不是检索本身，所以我把评测改成了 label-aware 的方式，按 relevance level 重新计算。这个过程让我意识到，RAG 实验里评测口径本身和检索模型同样重要。

这段很值钱，因为它体现的是：

- 你真的做了实验
- 你不是机械跑脚本
- 你能发现实验设计本身的问题

### Q3：你的 RAG 链路完整流程是什么？

推荐回答：

> 文档上传之后，我先做分片，再调用 embedding 模型生成向量，存到 Milvus。用户提问时，会先做向量检索拿到一批候选，再通过 rerank 模型做二次精排，最后只把少量高质量 chunk 和用户问题一起交给大模型生成答案。为了减少重复信息，我还加了单文档 chunk 上限，避免同一篇文档的相邻分片占满上下文。也就是说，RAG 这部分我做的是从入库到召回再到精排的完整链路，而不是只做了一个检索接口。

回答重点：

- 上传
- 分片
- 向量化
- 向量库存储
- 粗排检索
- rerank 精排
- 文档级限流
- 生成

### Q4：为什么要做多 Agent，而不是一个 Agent 搞定？

推荐回答：

> 因为 AIOps 排障是一个多步骤任务。它不是单轮问答，而是先规划，再执行，再根据执行结果调整下一步。如果一个 Agent 同时负责规划和执行，职责会比较混乱，也不容易控制流程。所以我把角色拆成了 Supervisor、Planner、Executor，分别负责调度、规划和执行，这样职责更清晰，也更适合后续扩展。

回答重点：

- 单 Agent 适合简单任务
- 多步骤排障更适合角色拆分
- 多 Agent 的核心不是“更高级”，而是“职责更清晰”

### Q5：你封装了哪些工具？模型怎么知道什么时候调用？

推荐回答：

回答重点：

- 工具要分能力边界
- 调用不是写死 if-else
- 但也不是完全放飞，还是通过 prompt 和描述来约束

### Q6：你说 `TopK 命中率 85%`，这个怎么来的？

推荐回答：

> 这个数据不是我拍脑袋说的，而是我基于现有知识库做的一版小规模离线检索实验。实验里我没有只放核心 runbook，还补了 FAQ、SOP、事故复盘和一些相似干扰文档，让它更接近真实知识库。query 也不是一种类型，我分成了关键词直问、口语化改写、步骤型问题和细节型问题。
>
> 评测时我主要看 `Hit@1`、`Hit@3` 和 `MRR`。其中 `Hit@3` 是最核心的，因为它能反映 Top3 里是否至少出现了一个相关文档。后来我又意识到，一个 query 并不一定只有一个“唯一正确文档”，有些问题其实更适合命中 FAQ 或 SOP，所以我把评测改成了 label-aware 的方式，给每个 query 标注强相关和弱相关文档，再分严格口径和宽松口径去算。  
>
> 所以我现在更稳的说法不是“我有一个绝对准确的 85%”，而是：我做过离线评测，并且把语料组织、query 设计、评分口径和对比组都做了拆分，能够比较清楚地判断检索优化到底有没有带来提升。

更安全的短版：

> 这是我基于现有知识库做的离线小样本评测结果，不是正式线上指标，但评测方法是成体系的。

回答重点：

- 一定强调 `离线实验`、`对比组`、`评分口径`
- 不要只报数字，要讲“怎么测的”
- 你要表现得像自己真的做过评测设计和复盘

### Q7：你为什么要区分核心文档和非核心文档？

回答重点：

- 核心文档是主 runbook
- 非核心文档不等于错误文档
- 你在区分“主文档路由”和“回答支持型检索”

### Q8：你为什么把 RAG 从一阶段检索升级成两阶段检索？

推荐回答：

> 因为我在实际调试时发现，纯向量 topK 更像是粗召回，能找到“可能相关”的内容，但不一定能把最适合回答问题的 chunk 排在最前面。尤其是在运维文档场景里，同一篇文档里相邻 chunk 很容易一起被召回，导致上下文里重复信息偏多。所以我把检索拆成了两阶段，先用 Milvus 做高召回粗排，再用 rerank 模型做精排，最后再通过单文档上限控制冗余。

回答重点：

- 一阶段向量检索不是错，但排序不够细
- 两阶段的目标是“排准”，不是只为了堆复杂度
- 你是根据实际问题迭代出来的，不是照着论文堆架构

### Q9：你做实验时有没有踩过坑？后来怎么修正的？

推荐回答：

回答重点：

- 主动承认实验设计踩坑
- 说明你能复盘并修正口径
- 最后落到“效果提升 + 延迟代价”这个更真实的结论

### Q10：为什么还要加 `per-doc-cap` 这种单文档上限？

回答重点：

- 目标是提升上下文覆盖面

### Q15：你这个多 Agent 到底是怎么编排的？

推荐回答：

> 我现在这套多 Agent 不是让多个 Agent 自由对话，而是 Supervisor 驱动的状态机式编排。具体来说，我把角色拆成了 Planner、Executor 和 Supervisor。Planner 负责拆解任务和再规划，Executor 只负责执行当前第一步并返回结构化反馈，Supervisor 不直接做具体分析，而是根据共享状态决定下一步该调 Planner、调 Executor，还是直接结束流程。  
>
> 在实现上，Planner 和 Executor 都是 ReactAgent，本质上各自内部还是 ReAct 循环，也就是模型决定要不要调工具、读取工具结果、再继续下一轮思考；Supervisor 则是更高一层的 FlowAgent，它负责在 sub-agent 之间做路由和回环控制。这样做的好处是角色边界比较清晰，也更容易控制失败条件和终止逻辑，比如我现在就限制了连续多次失败要及时终止，而不是让 agent 无限兜圈子。  
>
> 如果后面继续优化，我会把 Planner 和 Executor 之间传递的状态从自由文本升级成更强约束的结构化 state，比如明确 current_goal、current_step、tool_result、failure_reason，这样多 Agent 协作会更稳。

回答重点：

- 不是自由协商式 swarm，而是 Supervisor 驱动
- Planner 负责规划，Executor 负责执行，Supervisor 负责路由
- 关键价值是职责清晰、回环可控、失败可终止

如果面试官继续追问“为什么不用单 Agent”，可以补一句：

> 单 Agent 更适合简单问答，但 AIOps 是多步骤任务，既要规划又要执行，还要根据结果再调整。如果全部塞给一个 Agent，prompt 会很重，失败边界也不清楚；拆成角色之后，更像一个可控工作流。

### Q16：如果接入 PDF 文档，你会怎么做？

推荐回答：

> PDF 最难的地方不是向量化，而是版面噪声。因为 Markdown 天然有标题层级和段落边界，但 PDF 经常有页眉页脚、分页断裂、硬换行、表格和多栏布局，如果直接抽纯文本再切 chunk，检索效果通常会比较差。  
>
> 所以我觉得更合理的处理链路应该是四步。第一步先区分文本型 PDF 和扫描型 PDF，文本型直接提取文本层，扫描型要先做 OCR。第二步做版面清洗，比如去页眉页脚、合并断行、处理分页。第三步尽量重建结构，比如标题、章节、列表和页码。第四步再做结构化分片，优先按 section 和 paragraph 切，最后才用 token window 兜底。  
>
> 入库时我还会保留更丰富的 metadata，比如 file_name、page_no、section_title、source_type 和 chunk_index。这样后面不仅检索更稳，最终回答也更容易带来源引用。  
>
> 如果我在这个项目里继续做 PDF，我不会简单把它塞进当前 Markdown 分片逻辑，而是会把文档解析和分片策略拆成按类型扩展的 pipeline。

回答重点：

- PDF 的核心问题是版面噪声，不是 embedding
- 正确链路是：提取/OCR -> 清洗 -> 结构重建 -> 分片 -> 入库
- PDF 需要 richer metadata，不能只存 content

如果面试官继续追问“为什么不能 OCR 完直接分片”，可以补一句：

> 因为 OCR 只是把图片变成字，不能解决结构丢失问题。真正影响 RAG 效果的往往是标题、段落、表格这些信息是否还能被恢复出来。

### Q17：你现在的会话记忆怎么做，后面还能怎么优化？

推荐回答：

> 当前版本我做的是 Redis 持久化加窗口恢复。也就是最近几轮对话保留原始消息，长历史在恢复时通过摘要压缩，避免把整段聊天记录都塞进 prompt。这个方案已经能支持多轮会话和跨刷新恢复，但它本质上还是基础版 memory。  
>
> 后面如果继续优化，我会把记忆拆成四层。第一层是短期记忆，也就是最近几轮原始对话，保证上下文连续性。第二层是滚动摘要，把超出窗口的历史压成 summary，但这个 summary 最好是增量维护，而不是每次恢复时现算。第三层是结构化事实记忆，比如当前排查的服务、地域、已经查过哪些工具、已经确认了哪些结论，这些信息比纯聊天记录更稳定。第四层是语义记忆，把历史对话里真正高价值的片段做 embedding，当前问题来了再检索相关历史，而不是盲目把全部历史塞进去。  
>
> 对这个项目来说，我觉得最值得先做的是结构化事实记忆。因为它最适合 AIOps 场景，能让系统记住当前正在排查哪个告警、已经验证过什么，这样多轮任务感会比单纯摘要强很多。

回答重点：

- 当前是 Redis 持久化 + 窗口恢复 + 摘要压缩
- 更成熟的方向是四层记忆，不是只堆更多历史消息
- 最值得先做的是结构化事实记忆

如果面试官继续追问“为什么摘要还不够”，可以补一句：

> 摘要适合压缩语义，但不适合保存明确状态。排障场景里很多信息本质上是结构化的，比如告警名、服务名、地域、已执行步骤、失败原因，这些如果只放在摘要里，后续轮次很容易丢。

## 风险表述与安全说法

| 原始说法 | 风险 | 更安全的说法 |
|---------|------|-------------|
| TopK 命中率达到 85% | 会被追问数据来源、样本规模、命中定义 | 基于离线小规模样本做的初步评测，TopK=3 下命中率约 85% |
| rerank 一定比纯向量检索好 | 会被追问实验设计、评分口径和延迟代价 | 在我当前的 label-aware 离线评测里，两阶段检索在效果上更好，但代价是延迟增加，并且会出现主文档偏移问题 |
| 多 Agent 就是多个模型自由协商 | 会被追问具体编排方式 | 当前是 Supervisor 驱动的状态机式编排，Planner/Executor 分工明确，不是自由协商式 swarm |
| PDF 处理就是提文本再切 chunk | 会被追问版面和结构问题 | PDF 的重点是版面清洗和结构重建，提取文本只是第一步 |
| 会话记忆就是把聊天记录全塞给模型 | 会被追问 token 成本和稳定性 | 当前是 Redis 持久化 + 窗口恢复，后续会拆成短期记忆、摘要记忆、事实记忆和语义记忆 |
| RAG 已经做得很完善 | 会被追问 chunk、索引、rerank、评测细节 | 当前已经做了两阶段检索，但整体仍然是持续迭代中的原型方案 |
| 将运维响应时间从小时级降低到分钟级 | 会被追问有没有严格线上对比 | 从原型验证角度看，能显著缩短排障前置信息收集和初步分析时间 |
| 结合历史工单生成建议 | 容易被追问工单来源和接入方式 | 结合内部文档、日志和监控信息生成处理建议 |
| 已经对接真实外部日志服务 | 容易被追问线上环境细节 | 设计上支持通过 MCP 接入外部日志服务，当前以原型验证和 mock 调试为主 |
| 做了一个 AI 项目 | 太空，不体现工程价值 | 做了一个围绕知识问答和告警排障的企业智能代理原型系统 |
